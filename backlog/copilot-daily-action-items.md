# Copilot Daily Action Items (WorkIQ Integration)

## Overview

Automatically generate daily action items from WorkIQ (emails, chats, meetings) and surface them on the Sales Buddy dashboard alongside existing engagement action items.

## How It Works

### Daily Sync Trigger

Two scenarios:

1. **App running at 6 AM local time** - scheduler fires the WorkIQ query, clears yesterday's copilot action items, creates new ones from the response
2. **App starts after 6 AM** - on startup, check `last_copilot_sync` in the database. If the last sync was before 6 AM today, run the sync immediately

Track `last_copilot_sync` as a DateTime in `UserPreference` (or a new lightweight table). Update it after every successful sync.

### WorkIQ Query

Prompt: `"Look through all my emails, chats, and meetings, and let me know the top three things I need to get done."`

- Uses existing `query_workiq()` infrastructure
- Takes ~60 seconds
- Returns markdown with numbered action items, context, and source links

### Response Parsing

Extract from the markdown response:
- **Title** - the bolded action item heading (e.g. "Provide cost comparison for Vantage PostgreSQL scaling")
- **Why it matters** - the context paragraph
- **What to do** - the action steps
- **Source links** - Teams/Outlook URLs

Store each as an action item in the existing `EngagementTask` model (or a new table - see below).

### Action Item Lifecycle

- Created at sync time, visible immediately on dashboard
- User can mark as complete (same UX as engagement action items)
- Uncompleted items are cleared at next day's sync (replaced with fresh ones)
- Completed items are cleared too (they served their purpose)

## Data Model Changes

### Option A: Extend EngagementTask

Add a `source` column to `EngagementTask`:
- `'engagement'` (default) - current behavior, linked to an engagement
- `'copilot'` - generated by WorkIQ daily sync
- `'project'` (future) - from project management primitive

This requires making `engagement_id` nullable for copilot tasks (no parent engagement).

Pros: Reuses existing action items card, task completion UX, and dashboard query
Cons: EngagementTask was designed around engagements - nullable FK is a code smell

### Option B: New CopilotActionItem Table

Separate table with: id, title, description, source_url, status (open/completed), created_date, completed_at

Pros: Clean separation, no FK gymnastics
Cons: Duplicate UI rendering, separate dashboard query

### Recommendation: Option A

The dashboard Action Items card already renders tasks beautifully. Adding a `source` column and making `engagement_id` nullable keeps the UX unified. The `source` field also future-proofs for the project primitive.

## UI Changes

### Dashboard Action Items Card

- Copilot action items render in the same list as engagement tasks
- Visual indicator: small badge or icon showing source (e.g. Copilot sparkle icon)
- Copilot items sort below engagement items (engagement tasks are human-created and higher priority)

### Action Item Completion

- Same click-to-complete UX as engagement tasks
- Completing a copilot item just marks it done locally (no writeback anywhere)

### Loading State

- If sync is running when user loads dashboard, show a subtle spinner on the action items card
- 60-second WorkIQ query should not block page load - run in background thread

## Implementation Steps

1. **DateTime audit first** - fix all UTC/local mixing before adding scheduled tasks
2. Add `source` column to EngagementTask model + migration (default 'engagement')
3. Make `engagement_id` nullable on EngagementTask
4. Add `last_copilot_sync` to UserPreference + migration
5. Build WorkIQ action item parser (extract title, description, links from markdown)
6. Build sync function: query WorkIQ, parse response, clear old items, create new ones
7. Add startup hook: check `last_copilot_sync`, trigger sync if stale
8. Add scheduler: 6 AM daily trigger (using APScheduler or threading.Timer loop)
9. Update dashboard query to include copilot action items
10. Add copilot source badge to action item rendering
11. Tests for parser, sync lifecycle, startup hook

## Dependencies

- DateTime audit must be completed first (consistent timezone handling for scheduling)
- WorkIQ CLI must be installed and authenticated (`npx @microsoft/workiq`)
- No gateway or Azure changes needed (WorkIQ runs locally)

## Open Questions

- Should we let users customize the prompt? (e.g. "focus on customer-facing items")
- Should we persist completed copilot items for historical reference, or truly delete them?
- Should the 6 AM trigger time be configurable?
- What if WorkIQ is unavailable (no internet, auth expired)? Silent fail + retry next time?
